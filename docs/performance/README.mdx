# Performance cheats

---

## The goal

Most slow queries happen because they:

- Scan too many rows
- Sort too many rows
- Repeat the same work many times
- Prevent index usage

Fixing the query shape usually gives the biggest improvement.

## Why these hurt more in Google Cloud SQL

In self-managed PostgreSQL you can sometimes compensate with server tuning.

In Cloud SQL, you have less control over:

- Disk configuration
- I/O scheduling
- OS tuning
- Memory internals

So inefficient queries quickly cause:

- I/O saturation
- Temp file spills
- CPU bottlenecks
- Larger instance requirements

Which means higher cost and worse performance.

---

## Mental model for tuning

When a PostgreSQL query is slow, ask:

1. Can I avoid scanning rows I do not need?
2. Can I avoid sorting rows I do not need?
3. Can I avoid repeating the same work per row?
4. Can I make the predicate index-friendly?
5. Can I reduce the dataset before joining?

If the answer is yes to even one of those, there is often a big win available.

---

## Quick checklist

When reviewing a system, ask:

- Are queries using `OFFSET` pagination?
- Are indexed columns wrapped in functions?
- Do indexes match `WHERE + ORDER BY + LIMIT`?
- Are there correlated sub-queries?
- Are large joins happening before filtering?
- Are queries selecting unnecessary columns?
- Is the query indexing the right columns?
- Is the query avoiding unnecessary `OR` predicates?

Fixing just one of these issues often explains most performance problems.
